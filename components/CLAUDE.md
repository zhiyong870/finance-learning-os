# components/ 目录工作规则

本目录存放项目的六大核心组件。在本目录下工作时,本文件规则**叠加**在根 CLAUDE.md 之上。

---

## 1. 本目录的使命

本目录的每个子目录是一个独立学习组件。组件的存在价值,**不是**"提供一个功能",
而是"**创造一种让用户学到东西的交互**"。

### 六大组件及其学习目的

| 组件 | 学习目的 |
|---|---|
| `earnings_reader/` | 让用户真正读懂年报,而不是"知道年报说了啥" |
| `concept_trainer/` | 让用户内化金融概念,从"认识"到"拥有" |
| `reports_archive/` | 降低学习摩擦,让用户想读什么 5 秒能打开 |
| `case_library/` | 建立用户的模式识别能力,将单点经验连成网络 |
| `trade_journal/` | 让市场的教训真正转化为用户的能力 |
| `reading_queue/` | 让用户 18 周不迷失学习方向 |

**实现任何组件前,先问自己:这个组件如何服务它的学习目的?**

## 2. 组件必备文件

每个组件子目录必须包含:

```
components/<component_name>/
├── README.md            # 说明组件的学习目的、使用方式、设计理念
├── __init__.py
├── <main_module>.py     # 组件主逻辑
└── ...
```

**README.md 必须包含两节**:
- **学习目的**:这个组件如何让用户学到东西(具体到"用户会做什么动作、产出什么、学到什么")
- **反替思考论证**:这个组件在哪些环节强制用户思考,而不是让 AI 代劳

没有这两节的组件禁止合并。

## 3. 交互设计铁律

所有涉及 AI 的组件交互,必须遵循"**先你写,再 AI 反馈**"模式:

```
系统呈现原材料(年报段落 / 概念 / 案例 / 数据等)
    ↓
系统提示需要用户思考的问题
    ↓
【关键】用户先写出自己的理解或答案
    ↓
AI 读用户的答案 + 原材料,给反馈
    ↓
AI 追问 2-3 个深化问题
    ↓
用户继续写 → 循环
    ↓
会话结束,生成结构化的学习沉淀文件
```

**禁止的模式**:
- 系统直接给答案 / 总结 / 分析
- 用户问"这个公司怎么样",AI 直接给出评价
- 跳过"用户先写"步骤,直接进入 AI 输出

**组件里的 AI 交互必须加载 `.claude/skills/teaching-style.md`**,方式详见 tools/CLAUDE.md 中的 llm_wrapper 用法。

## 4. 学习沉淀规范

每次用户使用组件的会话结束后,必须生成**结构化沉淀文件**:

```markdown
---
component: earnings_reader
session_date: 2026-05-15
duration_minutes: 87
target: 贵州茅台 2023 年报 第三章
---

## 用户的理解演进

- 初始理解: ...
- 经过 AI 反馈后的修正: ...
- 最终理解: ...

## 做对的部分

...

## 遗漏或误解的部分

...

## 新提取的概念(进入 concepts/)

概念 A, 概念 B

## 新提取的案例(进入 cases/)

案例 C

## 未解决的疑问(进入 knowledge_gaps.md)

...
```

具体字段各组件可调整,但**沉淀必须是强制的,不是可选的**。

## 5. 组件间的耦合规则

- 组件尽量独立,通过 Markdown 文件系统交换信息,不直接互相调用
- 例:`earnings_reader` 产出的概念写入 `concepts/`,`concept_trainer` 扫描 `concepts/` 作为输入
- **禁止**组件 A 直接 import 组件 B 的内部模块
- 允许共享的东西放到 `tools/` 或 `components/_shared/`

## 6. 组件开发流程

新增或修改组件时,严格按此顺序:

1. **先写 README.md**:说清楚学习目的和反替思考论证,用户 review
2. **再设计交互流程**:用伪代码或流程图描述"用户在终端看到什么、做什么",用户 review
3. **设计数据结构**:沉淀文件 schema、状态管理
4. **实现核心逻辑**
5. **写端到端测试**(详见 tests/CLAUDE.md)
6. **实际跑一次**:用户亲自跑一次真实场景,反馈痛点
7. **迭代修改**

**严禁跳过第 1、2 步直接写代码。**

## 7. 本目录常见反模式警示

以下反模式在本目录高发,一旦出现必须打回:

- ❌ 函数名含 `auto_*`、`summarize_*`、`recommend_*`、`evaluate_*` 并直接输出结论
- ❌ Prompt 中出现"请总结"、"请推荐"、"请评估是否值得投资"
- ❌ AI 先输出答案,再"让用户补充",这不是"先你写"
- ❌ 交互流程里用户的输入是点击"下一步",没有实质输入
- ❌ 组件产生了炫酷输出但没有沉淀文件
- ❌ 组件 README 里讲了功能但没讲学习目的

**看到以上任一模式,停下来重新设计,不要"先实现再优化"。**

## 8. 和 tools/ 的协作

- 所有 AI 调用通过 `from tools.llm_wrapper import call_llm`
- 所有 prompt 放在 `tools/prompts/<component_name>/*.md`,不在代码里硬编码
- 所有 skill 加载通过 `tools/llm_wrapper` 的参数指定,不在组件里手动拼 prompt
