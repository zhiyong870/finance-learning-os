# 金融学习操作系统 — 项目工作规则

本文件是 Claude Code 在本项目工作时的**全局规则**,每次工作自动加载。
子目录有独立的 CLAUDE.md,补充该领域的具体规则,不重复本文件内容。

---

## 1. 项目使命(一切判断的最终依据)

**这是一个学习系统,不是投资系统,也不是交易系统。**

本项目的唯一目的:帮助用户(一名金融基础薄弱但研发能力强的开发者)在 18 周内,
从金融 Level 0-1 跃升至 Level 3,具备独立阅读年报、识别量化陷阱、用金融语言
表达投资逻辑的能力。

**任何功能、任何设计,如果不能直接服务于"用户自己学到东西",就要被砍掉。**
技术优雅、功能丰富、界面美观都不是本项目的追求。

## 2. 不可妥协的五条核心原则

### 2.1 工具必须逼用户思考,不能替用户思考
- **好的交互**:给用户原材料 → 提示关键点 → 让用户先写自己的理解 → AI 给反馈 → AI 追问
- **坏的交互**:一键总结、一键分析、一键推荐、一键评分
- 一切"一键 XX"的功能都是学习的敌人

### 2.2 AI 扮演老师,不是答题机
- **好的反馈**:"你的分析对了 70%,这 3 点是你漏掉的,为什么?"
- **坏的反馈**:"这家公司的投资价值是……"
- 具体规范见 `.claude/skills/teaching-style.md`

### 2.3 所有学习产出必须沉淀
- 每次精读、思考、错误、概念,都进入可检索、可复用、可回顾的知识库
- 不允许只有对话,没有沉淀文件
- 默认沉淀格式:Markdown,按规范命名,放在对应目录

### 2.4 强制性间隔重复
- 金融概念密度高,学后忘率超 70%
- 系统必须能主动考用户,不是被动等查
- 所有学过的概念进入 `concepts/` 目录,接受定期考察

### 2.5 Keep It Simple,早用重用
- 一个被每天用 3 次的简单功能,远强于 10 个被用过 1 次的华丽功能
- 不追求功能完整,只追求每个功能真的在用
- 宁可手动 5 分钟,不要自动化 2 小时去省 5 分钟

## 3. 技术栈(定稿,不随意扩展)

- **语言**:Python 3.11+
- **AI**:Claude API(通过 `tools/llm_wrapper.py` 统一调用)
- **存储**:Markdown 文件为主,SQLite 用于间隔重复等结构化数据
- **知识管理**:Obsidian vault(以项目根目录为 vault 根)
- **测试**:pytest
- **依赖管理**:uv 或 venv + requirements.txt
- **版本控制**:git,私有 GitHub 仓库

**引入新依赖前必须问用户。**能用标准库和现有依赖解决的,不引入新依赖。

## 4. 硬性禁区(绝对不做)

以下功能看似合理,但均违背项目使命,**任何情况下不实现**:

1. 实时股价监控、行情仪表盘 — 学习期看行情是最大杀手
2. 自动选股、荐股、打分 — 用户不懂,AI 也不靠谱,双重灾难
3. 技术指标分析(K 线形态、MACD、RSI 等) — 初学阶段走偏陷阱
4. 多用户 / 登录 / 权限系统 — 单用户个人工具
5. 漂亮的 Web UI、移动端 APP — 命令行 + Markdown 够用
6. 社交、分享、评论功能 — 不是产品
7. 模拟交易撮合引擎 — 用同花顺模拟盘就行

**如果用户提出做上述任一项,先引用本条目,请用户说明为什么例外,再继续。**

## 5. Workflow 规则

### 5.1 任务接收
- 每个任务开始前,先复述你对任务的理解给用户确认,再动手
- 任务描述不清或有歧义,**停下来问,不要猜**
- 任务涉及金融语义的假设,**必须问用户**(用户金融基础薄弱,但只有他知道学习目的)

### 5.2 任务执行
- 先写/改代码 → 再写/更新测试 → 跑测试 → 跑 invariants → 如适用调用 llm_review → 提交
- 测试没跑通不要声称任务完成
- 中途发现方案不对,**停下来重新确认**,不要硬推

### 5.3 结束前自检清单
完成一个任务前,逐条自检:
- [ ] 本次改动是否直接服务"用户学到东西"?
- [ ] 是否引入了"一键 XX"式的反模式?
- [ ] 所有 AI 调用是否都经过 `tools/llm_wrapper.py`?
- [ ] 是否有硬编码的 API key 或敏感信息?
- [ ] 测试是否覆盖核心路径?
- [ ] 新增文件是否放在正确目录?
- [ ] 涉及金融语义的地方,有没有凭记忆生成数字?

### 5.4 向用户汇报
- 用简洁的自然语言描述:做了什么、有什么产出、哪里需要用户确认
- 不做过度的自我表扬和冗长总结
- 有不确定的地方诚实说出,不粉饰

│   ├── llm_wrapper.py     # 所有 AI 调用必经

│   ├── invariants/        # Mechanical invariants 检查

├── components/            # 六大核心组件,详见 components/CLAUDE.md
│   └── CLAUDE.md
├── tests/                 # 测试,详见 tests/CLAUDE.md
│   └── CLAUDE.md

## 7. 和用户协作的约定

- 用户是项目拥有者、金融学习者、战略决策者
- 用户金融基础薄弱,涉及金融判断时**绝对不要代为决策**,要让用户自己判断
- 用户技术能力强,技术实现细节可以讨论也可以直接做,但方向性决策要确认
- 用户中文沟通,回复默认中文
- 用户时间宝贵,不要让他 review 不必要的细节

## 8. 违反本规则的后果

如果用户发现 Claude Code 违反了本规则,尤其是第 2、4 节,这是**严重问题**,
需要立刻停下来,说明原因,并更新 CLAUDE.md 或对应 Skills 避免复发。

**本文件是活的,会持续演进。每周复盘后可能更新。**
